# AI团队负责人推进表 SOP

## 用途

用于课程现场演示：如何把"很多人在参与"升级成"有人总推进、有人分板块、有人盯节奏、有人看反馈"的真实推进结构。

## 适用对象

老板/创始人/一号位、品牌/市场负责人、电商/运营负责人、内容团队、培训讲师。

## 核心目标

解决以下问题：
- "谁都在做一点但没有总推进人"
- "板块负责人不清楚"
- "当前阶段目标不清楚"
- "推进节奏和汇报机制不清楚"

## 核心认知（执行前必读）

执行本SOP前，先向用户传递以下3个关键认知：

1. **不是谁都负责，而是必须有人明确推进** — 只要责任一模糊，项目很快就会重新散掉。
2. **团队负责人推进表不是岗位通讯录，而是阶段推进结构图** — 关键不是"谁是谁"，而是"谁负责当前最重要的事"。
3. **没有推进节奏和催进度机制，再好的分工也会失效** — 所以推进表一定要把节奏和机制一起写进去。

## 一句话总结

> 不是谁都负责，而是必须有人总推进、有人分板块、有人盯节奏、有人看反馈。

## SOP主流程（17步）

按以下步骤执行，每步完成后输出对应结果。使用结构化表格呈现每一步的输出。

### 阶段0：明确本次任务

**任务目标**：统一认知：这一步不是简单写岗位，而是搭建真实推进结构。

**具体动作**：
- 明确目标：围绕回店后的执行，形成一张真实可用的《团队负责人推进表》

**需要收集的输入**：
- 品牌名
- 参课成员
- 课程主题
- 项目目标

**输出**：推进表任务定义

**注意事项**：开场要强调——不是谁都负责，而是必须有人明确推进。

---

### 阶段1：回收前面所有执行成果

**任务目标**：先把前面已经输出的成果接回来。

**具体动作**：
- 汇总以下成果：地图最终版、30天动作表、7天优先执行清单、三角色分工表、甘特表、验收表等

**需要收集的输入**：
- 各类表单、执行清单、节奏表、分工表

**输出**：推进输入底表

**注意事项**：推进表不是孤立文件，而是所有执行成果的组织承接表。如果用户还没有前面的成果，说明这些是前置条件，先补再做。

---

### 阶段2：明确总推进人

**任务目标**：找出真正对整体结果负责的人。

**具体动作**：
- 让AI帮助判断：谁最适合担任本项目的总推进人

**判断依据**：参课成员、岗位职责、组织结构、项目重要度

**输出格式**：

```
【总推进人确认表】

| 总推进人 | 为什么是他/她 | 核心职责 |
|---------|-------------|---------|
| XXX     |             |         |
```

**注意事项**：这一格最关键，不能空着，不能写成"大家一起"。

---

### 阶段3：划分核心板块

**任务目标**：先明确这次项目最关键的推进板块有哪些。

**具体动作**：
- 至少拆出五大板块：内容、搜索/电商、店铺承接、活动/直播、数据反馈

**需要收集的输入**：当前项目动作、组织结构、执行链路

**输出格式**：

```
【板块拆分表】

| 板块 | 说明 | 核心任务 |
|------|------|---------|
| 内容 |      |         |
| 搜索/电商 |  |         |
| 店铺承接 |  |         |
| 活动/直播 |  |         |
| 数据反馈 |  |         |
```

**注意事项**：板块不是越多越好，而是要对应真实推进结构。

---

### 阶段4：确认内容负责人

**任务目标**：明确谁负责内容板块推进。

**具体动作**：
- 指定内容负责人，并明确其当前阶段要负责的内容方向、节奏和交付

**需要收集的输入**：内容动作表、内容负责人名单、岗位分工

**输出格式**：

```
【内容负责人确认表】

| 内容负责人 | 当前阶段内容方向 | 关键交付物 | 时间节点 |
|-----------|---------------|----------|---------|
| XXX       |               |          |         |
```

**注意事项**：不能只写"内容团队"，要写到真实人。

---

### 阶段5：确认搜索/电商负责人

**任务目标**：明确谁负责站内搜索与电商承接推进。

**具体动作**：
- 指定搜索/电商负责人，并明确其当前阶段的重点任务

**需要收集的输入**：搜索承接表、站内优化表、电商岗位名单

**输出格式**：

```
【搜索/电商负责人确认表】

| 搜索/电商负责人 | 当前阶段重点任务 | 关键指标 | 时间节点 |
|--------------|---------------|---------|---------|
| XXX          |               |         |         |
```

**注意事项**：这一角色是承接内容与站内闭环的关键人。

---

### 阶段6：确认店铺承接负责人

**任务目标**：明确谁负责商品页、店铺页、活动页等承接动作。

**具体动作**：
- 指定店铺承接负责人，并明确其当前阶段最重要的页面优化目标

**需要收集的输入**：店铺承接优化表、商品页/店铺页动作、人员结构

**输出格式**：

```
【店铺承接负责人确认表】

| 店铺承接负责人 | 当前阶段优化目标 | 关键页面 | 时间节点 |
|-------------|---------------|---------|---------|
| XXX         |               |         |         |
```

**注意事项**：很多企业会把这块和电商负责人混在一起，这里要明确到人。

---

### 阶段7：确认活动/直播负责人

**任务目标**：明确谁负责活动页、直播间和节奏联动。

**具体动作**：
- 指定活动/直播负责人，并明确其阶段目标和配合事项

**需要收集的输入**：活动排期、直播安排、节点表、负责人名单

**输出格式**：

```
【活动/直播负责人确认表】

| 活动/直播负责人 | 阶段目标 | 配合事项 | 时间节点 |
|-------------|---------|---------|---------|
| XXX         |         |         |         |
```

**注意事项**：这块经常在大促期成为关键收口点，必须单独拎出来。

---

### 阶段8：确认数据反馈负责人

**任务目标**：明确谁负责收集和反馈阶段结果。

**具体动作**：
- 指定数据反馈负责人，并明确其负责回收哪些数据、在哪些节点反馈

**需要收集的输入**：搜索回流表、经营判断表、数据看板、复盘节奏

**输出格式**：

```
【数据反馈负责人确认表】

| 数据反馈负责人 | 回收数据类型 | 反馈节点 | 反馈对象 |
|-------------|-----------|---------|---------|
| XXX         |           |         |         |
```

**注意事项**：如果没人负责反馈，前面所有动作都很难形成闭环。

---

### 阶段9：明确每个人当前阶段最重要目标

**任务目标**：让负责人不只是有"岗位"，还有"当前任务"。

**具体动作**：
- 为每个负责人梳理当前阶段的关键目标，而不是泛职责

**需要收集的输入**：板块职责、阶段任务、30天动作表、7天清单

**输出格式**：

```
【负责人阶段目标表】

| 负责人 | 角色 | 当前阶段最重要目标 | 完成标准 | 截止时间 |
|-------|------|-----------------|---------|---------|
| XXX   | 总推进人 |               |         |         |
| XXX   | 内容负责人 |               |         |         |
| ...   |      |                 |         |         |
```

**注意事项**：这一步要写成"当前最重要目标"，不能写成长期宽泛职责。

---

### 阶段10：判断板块之间的依赖关系

**任务目标**：帮团队看清谁先动、谁后接。

**具体动作**：
- 梳理不同板块之间的先后关系和依赖逻辑

**需要收集的输入**：各板块任务、动作先后顺序、时间节点

**输出格式**：

```
【板块依赖关系表】

| 前置板块 | 后置板块 | 依赖关系说明 | 时间差 |
|---------|---------|-----------|-------|
| 内容     | 搜索/电商 |           |       |
| 搜索/电商 | 店铺承接 |           |       |
| ...     |         |           |       |
```

**注意事项**：没有依赖关系，就容易各板块自己跑自己的。

---

### 阶段11：设计推进节奏

**任务目标**：让推进表不只是静态分工，而是动态推进机制。

**具体动作**：
- 明确是按周推进、按阶段推进，还是按大促节点推进

**需要收集的输入**：节奏表、甘特表、项目时间、节点计划

**输出**：推进节奏表（参见 `references/rhythm-template.md`）

**注意事项**：推进节奏决定团队怎么开会、怎么复盘、怎么催动作。

---

### 阶段12：明确谁来催进度

**任务目标**：防止表填完后没有人推动。

**具体动作**：
- 指定谁来盯进度、催节点、拉复盘
- 通常是总推进人或指定项目管理角色

**需要收集的输入**：组织结构、总推进人、协同机制、周会机制

**输出格式**：

```
【催进度责任表】

| 催进度人 | 催什么 | 频率 | 催谁 | 方式 |
|---------|-------|------|-----|------|
| XXX     |       |      |     |      |
```

**注意事项**：没有催进度的人，推进表很容易沦为摆设。

---

### 阶段13：明确谁来做阶段汇报

**任务目标**：让负责人推进结果能被上收。

**具体动作**：
- 指定谁来做阶段汇报、汇报什么、向谁汇报

**需要收集的输入**：阶段目标、老板角色、复盘机制

**输出格式**：

```
【阶段汇报机制表】

| 汇报人 | 汇报内容 | 向谁汇报 | 频率 | 形式 |
|-------|---------|---------|------|------|
| XXX   |         |         |      |      |
```

**注意事项**：这一步决定组织内部是否真的形成闭环。

---

### 阶段14：形成《团队负责人推进表》初版

**任务目标**：把角色、目标、节奏真正整合成正式版成果。

**具体动作**：
- 将总推进人、板块负责人、阶段目标、推进节奏整合成一张表

**需要收集的输入**：前面所有确认结果

**输出格式**：参见 `assets/promotion-table-template.md`（团队负责人推进表模板）

**注意事项**：这是本SOP的核心输出。

---

### 阶段15：判断当前最关键的推进风险

**任务目标**：提前识别最可能出问题的地方。

**具体动作**：
- 判断当前推进结构中最可能出现的断点：
  - 没人拍板
  - 协同不顺
  - 负责人太分散
  - 时间节点不清晰
  - 反馈机制缺失

**需要收集的输入**：推进表初版、角色结构、动作链路

**输出格式**：

```
【推进风险识别表】

| 风险类型 | 风险描述 | 影响程度 | 应对方案 | 责任人 |
|---------|---------|---------|---------|-------|
|         |         |         |         |       |
```

**注意事项**：这一步是为了避免表做完以后，回去还是散。

---

### 阶段16：分配到真实回店成员

**任务目标**：确保表里不是虚角色，而是回去真能执行的人。

**具体动作**：
- 用真实参课成员和真实回店负责人填表，不写虚角色

**需要收集的输入**：真实姓名、岗位、组织架构、回店执行结构

**输出**：真实负责人清单

```
【真实负责人清单】

| 角色 | 姓名 | 岗位 | 是否回店后可执行 | 备注 |
|------|------|------|---------------|------|
| 总推进人 |   |   | ✅/❌ |   |
| 内容负责人 | |   | ✅/❌ |   |
| 搜索/电商负责人 | |   | ✅/❌ |   |
| 店铺承接负责人 | |   | ✅/❌ |   |
| 活动/直播负责人 | |   | ✅/❌ |   |
| 数据反馈负责人 | |   | ✅/❌ |   |
```

**注意事项**：这一步一定要按真实推进结构填，不然没意义。

---

### 阶段17：建立滚动更新机制

**任务目标**：让推进表随着节奏变化而调整。

**具体动作**：
- 设计后续每周/每阶段如何更新推进表

**需要收集的输入**：推进节奏、周会节奏、阶段目标、复盘机制

**输出格式**：

```
【推进表滚动更新机制表】

| 更新项 | 更新频率 | 由谁更新 | 更新触发条件 | 反馈方式 |
|-------|---------|---------|-----------|---------|
|       |         |         |           |         |
```

**注意事项**：推进表不是一次性分工表，而是持续更新的组织推进底图。

## 标准理解参考

执行SOP时参照五个标准模块：

| 模块 | 它回答的问题 | 解释 |
|------|------------|------|
| 总推进人 | 谁对整体推进真正负责 | 不是谁都负责，而是必须有一个清晰推进人 |
| 各板块负责人 | 各个模块分别谁来扛 | 内容、搜索/电商、店铺承接、活动/直播、数据反馈都要有人负责 |
| 当前阶段目标 | 负责人现在最重要的任务是什么 | 不是泛职责，而是当前阶段的关键目标 |
| 推进节奏 | 这套推进结构按什么节奏跑 | 按周推进、按阶段推进，还是按大促节点推进 |
| 催进度与汇报 | 谁来拉着这件事往前走 | 推进机制决定这张表能不能真正活起来 |

```
【老板版摘要】

一句话结论：不是谁都负责，而是必须有人总推进、有人分板块、有人盯节奏、有人看反馈。

谁是总推进人：XXX
各板块负责人分别是谁：内容-XXX / 搜索电商-XXX / 店铺承接-XXX / 活动直播-XXX / 数据反馈-XXX
每个人当前阶段最重要目标是什么：（逐一列出）
当前按什么节奏推进：按周/按阶段/按大促节点
谁来催进度，谁来汇报：XXX催进度 / XXX向XXX汇报
```
