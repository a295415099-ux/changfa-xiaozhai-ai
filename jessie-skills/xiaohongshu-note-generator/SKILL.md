# 小红书笔记生成 Skill

核心原则：小红书笔记不是商品详情页，不是产品介绍、官方卖点、活动说明或详情页压缩版。要写成「一个人，在一个具体场景下，遇到一个真实问题，找到一个可参考的解决方案」。核心不是「夸产品」，而是让用户觉得：你懂我的场景，你说中了我的问题，你给了我一个可以参考的选择。

## 执行前必读

**在开始执行之前，必须先确认以下信息：**

- 品牌名 / 产品名
- 目标人群（谁在看这篇笔记）
- 使用场景（用户在什么情况下使用）
- 用户痛点（用户遇到什么问题）
- 核心卖点（产品解决什么问题）
- 关键词（用户会搜什么词）
- 淘内承接位置（商品页 / 直播间 / 客服入口）

如果用户没有提供以上信息，必须先通过 interview 收集。使用选择题格式，最多5个问题一轮，最多2轮。

## 执行流程（20阶段完整SOP）

按以下阶段顺序执行，每完成一个阶段，输出该阶段的产出物，然后进入下一阶段。不要跳过任何阶段。

---

### 阶段0：明确本次任务

**任务目标**：统一认知——这不是写详情页缩写版，而是生成种草笔记结构。

**动作**：
- 明确本次输出目标：生成《小红书笔记模板》及标题、开头、正文结构、评论区引导、承接建议
- 向用户确认：小红书内容核心是让用户产生代入与参考感

**输入**：品牌名、课程主题、产品资料、内容目标

**输出**：笔记生成任务定义

**注意事项**：开场强调小红书内容不是广告、不是产品说明书

---

### 阶段1：回收基础资料

**任务目标**：先把笔记生成所需底层信息拉出来。

**动作**：汇总以下信息：
- 品牌资料
- 产品信息
- 目标人群画像
- 使用场景
- 用户痛点
- 核心卖点
- 关键词表
- 淘内承接位置（商品页/直播间/客服入口）

**输入**：品牌资料、商品资料、用户画像、关键词表、商品页

**输出**：笔记基础信息表（结构化列出以上所有信息）

**注意事项**：没有底层信息，AI 生成内容会流于空泛。如果用户无法提供完整信息，标注缺失项并说明影响。

---

### 阶段2：明确内容任务

**任务目标**：判断这篇小红书笔记主要承担什么任务。

**动作**：明确笔记更偏向哪种类型：
- 种草型（让用户产生兴趣和向往）
- 教育型（教用户如何选择/使用）
- 避坑型（帮用户避开常见问题）
- 选择建议型（帮用户在多个选项中做决策）
- 搜索影响型（抢占关键词搜索结果）
- 承接型（引导用户去淘内完成购买）

**输入**：内容目标、商品策略、平台定位

**输出**：笔记任务确认表（明确任务类型+原因）

**注意事项**：不同任务会影响标题和正文结构。

---

### 阶段3：生成标题方向

**任务目标**：让目标用户一眼知道这篇内容和自己有关。

**动作**：围绕以下7种标题类型，分别生成标题：
1. **人群型**：直接点名目标人群 → 「XX人群必看」「XX星人请进」
2. **场景型**：描述使用场景 → 「XX场景下」「每次XX都会」
3. **痛点型**：说出用户痛点 → 「XX问题终于解决了」「再也不用XX」
4. **避坑型**：帮用户避开坑 → 「XX千万别买」「XX避坑指南」
5. **清单型**：列出选项或方法 → 「XX个XX推荐」「XX清单」
6. **对比型**：对比不同选项 → 「XX vs XX」「XX和XX怎么选」
7. **关键词型**：直接踩搜索关键词 → 品类词+修饰词组合

每种类型至少生成1个标题方向。

**输入**：目标人群、场景、痛点、关键词

**输出**：标题方向库（7种类型各至少1条）

**注意事项**：标题不要像广告标题，要像用户会点开的标题。

---

### 阶段4：生成开头钩子

**任务目标**：让用户前3行产生代入感。

**动作**：围绕以下4种角度生成开头：
1. **用户状态型**：描述用户当下的状态或情绪 → 「最近是不是……」「你有没有发现……」
2. **用户痛点型**：直接说出痛点 → 「XX真的太让人崩溃了」
3. **真实场景型**：把用户放进一个场景 → 「上周在XX的时候……」「每次XX都会出现一个情况」
4. **为什么值得看型**：给出继续阅读的理由 → 「花了XX时间/XX钱总结出来的」「用了XX方法终于」

每种角度至少生成1条开头。

**输入**：痛点、场景、用户问题、真实状态

**输出**：开头钩子库（4种角度各至少1条）

**注意事项**：开头不是介绍产品，而是先建立代入感。「这说的不就是我吗」是正确反应。

---

### 阶段5：设计正文主体

**任务目标**：把内容组织成清晰可读的种草逻辑。

**动作**：按以下结构组织正文框架：
1. 痛点共鸣 → 描述用户遇到的问题
2. 常见误区 → 指出大家容易踩的坑
3. 解决思路 → 给出你的思考路径
4. 产品体验 → 用真实体验描述产品
5. 适合人群 → 说明谁适合/谁不适合
6. 使用建议 → 给出具体怎么用的建议

按照「痛点→误区→解决思路→产品体验→适合人群→使用建议」结构输出正文框架。

**输入**：场景、问题、选择标准、体验、结果

**输出**：正文结构表

**注意事项**：正文必须像真实经验分享，而不是品牌自述。不能只写产品好，要写出「为什么会选择它」「用过之后什么感受」。

---

### 阶段6：设计卖点转译

**任务目标**：把品牌卖点转成用户能感知的体验与价值。

**动作**：将每个产品卖点翻译成：
1. **用户感受**：用了之后是什么感觉
2. **使用体验**：使用过程中发生了什么改变
3. **场景价值**：在具体场景下帮用户解决了什么
4. **选择理由**：为什么选它而不是其他
5. **避坑建议**：什么情况下不适合选它

每个卖点对应一条转译结果。

**输入**：核心卖点、用户语言、使用体验、对比信息

**输出**：卖点转译表

**注意事项**：不能只写品牌卖点，要写「人话」。例如：「采用XX技术」→「用了之后皮肤再也不干了」。

---

### 阶段7：设计适合/不适合人群

**任务目标**：帮助用户更快判断这篇内容和商品是否适合自己，降低判断成本。

**动作**：
- 明确适合哪些人群（基于场景、需求、预算等维度）
- 明确不适合哪些人群（基于产品限制、需求不匹配等）
- 每类人群附带一句简短的判断理由

**输入**：目标人群、场景、产品限制、用户需求

**输出**：适合/不适合人群表

**注意事项**：这一步会直接提升内容可信度。敢于说「不适合谁」比只说「适合谁」更有说服力。

---

### 阶段8：设计关键词布局

**任务目标**：让用户记住并影响后续搜索。

**动作**：围绕以下6类关键词进行布局：
1. **品牌词**：品牌名
2. **品类词**：产品所属品类
3. **卖点词**：核心功能/优势相关词
4. **场景词**：使用场景相关词
5. **问题词**：用户痛点/问题相关词
6. **人群词**：目标人群相关词

说明每类关键词应该出现在笔记的哪个位置（标题/开头/正文/结尾）。

**输入**：关键词表、笔记结构、内容主题

**输出**：关键词布局表

**注意事项**：关键词要自然出现，不要堆砌。自然出现的关键词比硬塞更有效。

---

### 阶段9：设计评论区引导

**任务目标**：让评论区继续产生内容机会。

**动作**：围绕以下5个方向生成互动提问：
1. **痛点提问**：引出用户类似的痛点 → 「你们有没有遇到过XX？」
2. **场景提问**：引发场景共鸣 → 「你们一般XX的时候会怎么处理？」
3. **疑虑提问**：让用户说出顾虑 → 「你们最担心XX的什么？」
4. **购买问题**：引出购买相关讨论 → 「你们会选A还是B？」
5. **使用反馈**：让已购用户分享 → 「用过的姐妹来说说感受？」

每个方向至少生成2条互动提问。

**输入**：用户问题、常见疑虑、内容主题

**输出**：评论区引导表

**注意事项**：好的评论区引导会反哺选题和客服话术，形成内容飞轮。

---

### 阶段10：设计淘内承接建议

**任务目标**：让小红书笔记与淘内搜索和承接连起来。

**动作**：明确用户看完笔记后的下一步行动路径：
1. **搜什么**：明确告诉用户去淘宝搜什么关键词
2. **去哪里看**：指向具体的商品页/店铺页
3. **去哪里问**：指向直播间/客服入口
4. **看什么**：建议用户关注详情页的哪个部分

**输入**：商品标题、详情页、直播间、客服入口、关键词

**输出**：淘内承接建议表

**注意事项**：小红书不能只负责种草，还要帮助用户明确下一步行动。

---

### 阶段11：生成完整小红书笔记结构

**任务目标**：把标题、开头、正文、卖点、关键词、评论区整合成一篇完整笔记。

**动作**：基于 W+I→N 结构生成一篇完整小红书种草笔记，包含：
- 精选标题（从标题方向库中选最优的）
- 开头钩子
- 正文主体（按阶段5的结构）
- 卖点转译融入正文
- 适合/不适合人群
- 关键词自然融入
- 评论区引导文案
- 淘内承接引导

**输入**：品牌名、产品名、人群、场景、痛点、卖点、关键词、承接入口

**输出**：小红书笔记初稿（完整一篇）

**注意事项**：这是本 Skill 的核心生成环节。完整笔记应该读起来像一个真实用户写的分享。

---

### 阶段12：检查标题是否能点开

**任务目标**：验证标题是否像用户会点开的标题。

**检查清单**：
- [ ] 标题读起来像用户会搜索的内容吗？
- [ ] 标题像用户会收藏的标题吗？
- [ ] 标题像用户会点击的标题吗？
- [ ] 标题是否避免了品牌广告腔？
- [ ] 标题是否包含用户关心的关键词？

**输入**：标题方向、用户语言、平台风格

**输出**：标题点击性检查表

**注意事项**：如果标题像广告，打回去重写。

---

### 阶段13：检查开头是否建立代入

**任务目标**：验证用户看前3行是否会觉得「这就是我」。

**检查清单**：
- [ ] 开头是否说中了用户的状态？
- [ ] 开头是否说中了用户的问题？
- [ ] 开头是否说中了用户的场景？
- [ ] 前3行是否能让人产生「这说的不就是我吗」的感觉？

**输入**：开头钩子、用户痛点、场景

**输出**：开头代入感检查表

**注意事项**：开头没有代入，正文再完整也难继续读。

---

### 阶段14：检查正文是否有真实经验感

**任务目标**：验证正文是否像真实体验和选择逻辑。

**检查清单**：
- [ ] 正文是否写出了具体的场景？
- [ ] 正文是否写出了真实的问题？
- [ ] 正文是否写出了选择逻辑（为什么选它）？
- [ ] 正文是否写出了使用体验（用了之后怎样）？
- [ ] 正文是否避免了只写「产品好」？

**输入**：正文结构、卖点转译、体验表达

**输出**：正文经验感检查表

**注意事项**：真实经验感是小红书笔记和广告的关键区别。

---

### 阶段15：检查关键词是否自然

**任务目标**：验证关键词有没有自然进入内容。

**检查清单**：
- [ ] 品牌词是否自然出现在内容中？
- [ ] 品类词是否自然出现？
- [ ] 场景词是否融入描述？
- [ ] 问题词是否帮助记忆与搜索？
- [ ] 关键词没有堆砌感？

**输入**：关键词布局、正文内容、标题

**输出**：关键词自然度检查表

**注意事项**：关键词自然出现比硬塞更有效。

---

### 阶段16：检查评论区是否能引出用户问题

**任务目标**：验证评论区引导是否有利于后续内容机会。

**检查清单**：
- [ ] 评论区提问能否继续引出痛点讨论？
- [ ] 评论区提问能否引出场景共鸣？
- [ ] 评论区提问能否引出用户疑虑？
- [ ] 评论区提问能否引出购买相关问题？

**输入**：评论区引导、用户问题、客服问答

**输出**：评论区机会检查表

**注意事项**：评论区会反过来成为下一轮内容机会。

---

### 阶段17：形成《小红书笔记模板》初版

**任务目标**：把笔记结果沉淀成正式工具表。

**动作**：将以下内容整合成正式版模板：
- 标题
- 开头
- 正文结构
- 卖点转译
- 适合/不适合人群
- 关键词
- 评论区引导
- 淘内承接建议

**输入**：前面所有阶段的产出结果

**输出**：《小红书笔记模板》初版

**注意事项**：这是本 Skill 的核心正式输出。模板格式见 `assets/note-template.md`。

---

### 阶段18：输出当前最值得优先发的笔记

**任务目标**：帮团队知道先发哪篇。

**动作**：从全部笔记中筛出当前最值得优先执行的 1-3 篇，筛选标准：
- 与当前商品/营销目标最匹配
- 关键词搜索量最大
- 用户痛点最强烈
- 内容与竞品差异化最大

**输入**：笔记模板、商品目标、平台目标

**输出**：优先发布笔记清单（1-3篇，含排序理由）

**注意事项**：团队不要同时铺太多内容而失焦。

---

### 阶段19：分配到团队执行

**任务目标**：让笔记进入内容、运营、电商、客服团队使用。

**动作**：明确分工——
- 谁负责写（内容团队/小红书运营）
- 谁负责发（小红书运营）
- 谁负责评论区互动（客服/运营）
- 谁负责淘内承接（电商团队：商品页/直播间/客服）

**输入**：笔记模板、团队职责、业务流程

**输出**：团队执行分工表

**注意事项**：笔记不进入实际分工，就只是纸上作业。

---

### 阶段20：建立滚动复盘机制

**任务目标**：让笔记生成结果持续服务后续内容生产。

**动作**：设计后续复盘维度——
- 曝光数据（小眼睛数、展现量）
- 互动数据（点赞、收藏、评论、分享）
- 搜索影响（关键词搜索排名变化）
- 进店数据（从笔记到店铺的流量）
- 承接表现（店铺转化、直播间转化）

设定复看周期（建议每周/每两周），持续优化笔记方向。

**输入**：曝光数据、互动数据、搜索回流、进店数据

**输出**：笔记复盘机制表

**注意事项**：小红书内容要随着反馈不断优化，而不是发完就结束。
